# Comment 9 agents IA collaborent pour coder, tester et déployer automatiquement

**Le développement logiciel moderne est fractal.** Chaque feature, même modeste, touche 5 à 10 contextes simultanés : le modèle de données, la logique métier, le composant visuel, les tests, la documentation, le déploiement, la sécurité. Un développeur humain jongle avec ces contextes en permanence — mais un LLM seul, même très performant, peine à maintenir cette cohérence multi-domaine sur des tâches complexes. C'est le constat qui a motivé la création de Swarm.

## Le concept Swarm : 9 agents, un pipeline

Contrairement aux assistants mono-agents comme Claude Code, Cursor ou Devin, Swarm n'est pas un chatbot qu'on guide pas à pas. C'est un pipeline où **9 agents IA spécialisés** collaborent automatiquement, chacun avec un rôle précis et des contraintes strictes :

| Agent | Rôle |
|---|---|
| **Orchestrateur** | Point d'entrée unique, classification, routage, mémoire |
| **Search** | Exploration du codebase, grep parallèle, comptage de fichiers |
| **Planner** | Plan d'implémentation structuré, points de contrôle |
| **Contract** | Spécifications formelles (contrat d'interface) |
| **Front** | Composants UI, animations, expérience Apple-grade |
| **Back** | Logique métier, modèles, services |
| **Tester** | Tests unitaires (Jest) et E2E (Playwright), coverage ≥ 80% |
| **Reviewer** | Revue de code, sécurité, qualité, standards |
| **Writer** | Documentation, CHANGELOG, README |

Chaque agent a des **contraintes explicites**. L'orchestrateur ne code jamais. Le planner ne modifie pas de fichiers. Le front ne touche pas à la logique métier. Cette spécialisation radicale est la clé : elle réduit les hallucinations en limitant le contexte de chaque agent à son domaine d'expertise.

## L'orchestration : classification automatique en 5 routes

Le véritable défi n'est pas d'avoir 9 agents — c'est de **savoir lequel activer, quand, et dans quel ordre**. L'orchestrateur résout ce problème avec une **pre-search obligatoire** : avant toute classification, il extrait 2 à 5 termes techniques de la demande, lance des grep parallèles sur le codebase, et compte les fichiers distincts touchés par domaine.

Résultat : 5 routes de complexité croissante.

| Route | Fichiers touchés | Gates qualité |
|---|---|---|
| **DIRECT** | 0 | Aucune |
| **SIMPLE** | 1–2 | Test PASS |
| **ADAPT** | 2–4 | Test PASS |
| **MEDIUM** | 4+ | Test PASS + Reviewer APPROVE |
| **FULL** | Contrat + multi-domaine | Test PASS + Reviewer APPROVE |

Cette classification est automatique et déterministe — pas de prompt demandant au LLM d'estimer sa propre difficulté (ce qui est notoirement peu fiable). Le nombre de fichiers touchés, détecté par grep, est le critère objectif.

## Un exemple concret : une feature MEDIUM de bout en bout

Prenons une demande réelle : *« Ajoute une section "Dernières mises à jour" sur la homepage avec une timeline des commits récents »*.

**1. Pre-search (Orchestrateur + Search)**  
L'orchestrateur extrait les termes `timeline`, `commits`, `homepage`, `changelog`. L'agent search lance des grep parallèles et trouve 8 fichiers concernés : le composant homepage, le service de contenu, les modèles TypeScript, les tests unitaires, les tests E2E, le CHANGELOG, les routes, et le SCSS. La route est classée **MEDIUM** (4+ fichiers).

**2. Planification (Planner)**  
Le planner analyse les fichiers existants et produit un plan en 5 étapes avec points de contrôle : (1) modèle TypeScript pour les entrées de timeline, (2) service pour parser le CHANGELOG, (3) composant timeline avec animations GSAP, (4) tests unitaires Jest, (5) tests E2E Playwright.

**3. Implémentation parallèle (Front + Back)**  
L'orchestrateur délègue en parallèle : l'agent back crée le modèle et le service, l'agent front crée le composant visuel avec animations stagger. Chaque agent ne voit que les fichiers de son domaine — pas de pollution de contexte.

**4. Tests (Tester)**  
Le tester génère les tests Jest pour le service et les tests Playwright pour le flux utilisateur complet. Il vérifie le coverage (l'objectif est ≥ 80%) et les cas limites (CHANGELOG vide, format invalide, responsive mobile).

**5. Revue (Reviewer)**  
Le reviewer audite le code : sécurité des dépendances, respect de la palette dark exclusif, qualité des animations, cohérence du copywriting français. Score minimum requis : `security_score ≥ 1.0`, `quality_score ≥ 0.85`.

**6. Merge (Orchestrateur)**  
Toutes les gates validées (`tester_pass: true`, `reviewer_approved: true`), l'orchestrateur déclenche le merge via `finish.ts`. La PR est créée, les tests passent en CI, le déploiement Vercel se déclenche automatiquement.

Temps total : 8 minutes. Une tâche équivalente prendrait 2 à 4 heures à un développeur humain, entre la navigation dans le codebase, l'implémentation, les tests, et la revue.

## Résultats mesurables

Sur un échantillon de 20 tâches exécutées par la Swarm :

- **Coût API** : 5 à 8× inférieur à une approche mono-agent qui envoie l'intégralité du codebase dans chaque prompt. La spécialisation réduit le contexte par agent de 80%.
- **Qualité** : 194 tests unitaires, 16 specs E2E, coverage 89,5%. Les gates systématiques (test + review) éliminent les régressions silencieuses.
- **Traçabilité** : chaque exécution écrit `.agent-memory.json` et `.swarm-queue.json` — l'historique complet des décisions est versionné.
- **Déterminisme** : la classification par grep est reproductible. Même demande, même route, mêmes agents activés.

## Ce que la Swarm n'est pas

La Swarm n'est pas un SaaS. C'est un pipeline open source documenté en détail sur un wiki technique : [swarm-wiki.vercel.app](https://swarm-wiki.vercel.app). Le code est disponible sur [github.com/JohTandou/agent-swarm](https://github.com/JohTandou/agent-swarm).

La Swarm n'est pas non plus une solution magique. Elle excelle sur les tâches structurées (features, bugfixes, documentation) mais ne remplace pas la créativité humaine sur les décisions architecturales fondamentales. Les agents sont des exécutants spécialisés, pas des architectes.

## Et maintenant ?

Si vous construisez des pipelines agentic ou explorez l'automatisation du développement logiciel, le wiki documente l'architecture complète : les 9 agents, les 26 skills, les 5 routes, les outils MCP, et les standards de qualité. Tout est open source, en français, avec un design Apple-grade dark mode exclusif.
